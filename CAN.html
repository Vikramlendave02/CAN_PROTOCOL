CAN Protocol Fundamentals

Essential concepts and technical details of the Controller Area Network

Introduction & History

Developed by Bosch in the 1980s for automotive applications

  • 1983: First CAN project at Bosch
  • 1986: CAN protocol introduced
  • 1991: CAN 2.0A specification
  • 1993: ISO 11898 standard
  • Today: Used in 99% of modern vehicles

Key Features

Why CAN is the backbone of automotive communication

  • Multi-master architecture
  • 1 Mbps max speed (500kbps typical)
  • Robust error detection & handling
  • Differential signaling for noise immunity
  • Priority-based arbitration
  • Low cost implementation

Electrical Signaling

Differential voltage for noise immunity

State CAN_H CAN_L Differential Voltage
Dominant (0) 3.5V 1.5V 2V
Recessive (1) 2.5V 2.5V 0V

Frame Formats

Message structures in CAN communication

  • Data Frame: Transmits actual data (0-8 bytes)
  • Remote Frame: Requests data transmission
  • Error Frame: Signals detected errors
  • Overload Frame: Introduces delay between frames

Error Handling

Robust error detection and confinement

  • Error States: Active, Passive, Bus Off
  • Error Counters: TEC (Transmit), REC (Receive)
  • Automatic retransmission
  • Faulty node disconnection
  • Error types: CRC, ACK, Bit, Stuffing

Bit Timing

Configuring communication speed

Bit time = 1 / Baud rate

Segments:

  • Sync_Seg: 1 time quantum (fixed)
  • Prop_Seg: Compensates signal delay
  • Phase_Seg1/2: Resynchronization

Interview Questions & Answers

Common CAN protocol questions with detailed explanations

Explain CAN's non-destructive bit-wise arbitration

Mechanism: Nodes transmit dominant (0) and recessive (1) bits simultaneously. If a node transmits recessive but reads dominant, it loses arbitration and stops transmitting immediately.

Priority Handling: Lower 11/29-bit IDs win (e.g., ID 0x100 beats 0x200 since MSB 0 > 1).

Criticality: Prevents data collisions without time loss (unlike Ethernet CSMA/CD). Ensures high-priority messages (e.g., braking) access the bus first.

CAN Bus Arbitration Diagram
Describe CAN's error confinement states
State Condition Action
Error Active TEC/REC < 128 Sends active error flags (6 dominant 0s)
Error Passive TEC or REC ≥ 128 Sends passive error flags (6 recessive 1s)
Bus Off TEC > 255 Disconnects from bus

Counter Rules:

  • TEC (Transmit Error Counter): +8 for transmission errors (e.g., ACK error). +1 for passive/active error flag transmission.
  • REC (Receive Error Counter): +1 for detected errors during reception. +8 for passive/active error flag reception.
Compare Standard vs. Extended CAN frames
Feature Standard Frame Extended Frame
ID Length 11-bit 29-bit (11-base + 18-ext)
IDE Bit Dominant (0) Recessive (1)
Compatibility Supported by all 2.0B controllers only

IDE Bit Role: IDE transmitted after 11-bit ID. If a 2.0A node sees recessive IDE (illegal), it generates an error. Ensures backward compatibility: 2.0B nodes handle both frames.

How to configure bxCAN filters to accept only remote frames with ID 0x65D?
STM32 HAL Configuration Copy 
CAN_FilterTypeDef filter;
filter.FilterIdHigh = (0x65D << 5) | (CAN_ID_STD << 2); // 11-bit ID
filter.FilterMaskIdHigh = 0x7FF << 5; // Exact ID match
filter.FilterFIFOAssignment = CAN_FILTER_FIFO0;
filter.FilterMode = CAN_FILTERMODE_IDMASK;
filter.FilterScale = CAN_FILTERSCALE_32BIT;
filter.FilterActivation = ENABLE;
HAL_CAN_ConfigFilter(&hcan, &filter);

Critical Settings:

  • FilterMaskIdHigh: Set IDE/RTR bits to 1 (recessive) for remote frames
  • FilterIdHigh: RTR = 1 (bit 1) + IDE = 0 (bit 2) for standard remote frame
Calculate TSEG1/TSEG2 for 250kbps with 40MHz clock, prescaler=8, 75% sample point

Calculation Steps:

  1. CAN Clock: 40MHz / 8 = 5MHz
  2. Time Quantum (tq): 1 / 5MHz = 200ns
  3. Bit Time: 1 / 250kbps = 4μs
  4. Total tq: 4μs / 200ns = 20 tq
  5. Segments:
    • Sync_Seg = 1 tq (fixed)
    • Prop_Seg + Phase_Seg1 = 15 tq (75% of 20 tq)
    • Phase_Seg2 = 20 - 1 - 15 = 4 tq
    • TSEG1 = Prop_Seg + Phase_Seg1 = 15 tq
    • TSEG2 = Phase_Seg2 = 4 tq

STM32 bxCAN Implementation

Practical guide to using CAN on STM32 microcontrollers

STM32 bxCAN Features

  • Supports CAN protocol version 2.0 A/B
  • Bit rates up to 1 Mbit/s
  • Three transmit mailboxes
  • Two receive FIFOs with three stages each
  • 28 filter banks shared between CAN1 and CAN2
  • Master/slave configuration for dual CAN

Transmission Workflow

1

Select Empty Mailbox

Application selects an empty transmit mailbox

2

Configure Message

Set identifier, DLC (data length code), and data

3

Request Transmission

Set TXRQ bit in control register

4

Scheduler Priority

Mailbox waits to become highest priority

5

Bus Transmission

Operating Modes

  • Normal Mode: Regular communication
  • Loopback Mode: Self-test (TX internally looped to RX)
  • Silent Mode: Monitor bus without transmitting
  • Combined Modes: Silent + Loopback for internal testing
Transmitting a CAN Message Copy 
CAN_TxHeaderTypeDef TxHeader;
uint8_t TxData[8];
uint32_t TxMailbox;

TxHeader.StdId = 0x123;            // Standard identifier
TxHeader.ExtId = 0x00;              // Extended identifier (not used)
TxHeader.IDE = CAN_ID_STD;          // Standard ID type
TxHeader.RTR = CAN_RTR_DATA;        // Data frame
TxHeader.DLC = 8;                   // Data length (8 bytes)
TxHeader.TransmitGlobalTime = DISABLE;

// Prepare data
TxData[0] = 0x01;
TxData[1] = 0x02;
// ... fill all 8 bytes

// Add to transmit mailbox
if (HAL_CAN_AddTxMessage(&hcan, &TxHeader, TxData, &TxMailbox) != HAL_OK) {
    // Error handling
}
